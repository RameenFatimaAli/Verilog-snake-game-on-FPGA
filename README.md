# Snake Game on FPGA

A hardware implementation of the classic Snake game built entirely in Verilog HDL. The player controls a snake that grows longer each time it eats an apple, while avoiding the walls and its own body. All game logic, display output, and input handling are circuits synthesized onto a Cyclone V FPGA running at 50 MHz. The snake's position, apple placement, collision detection, and score are all computed in hardware logic gates running simultaneously and in parallel. Score is tracked up to 99 and shown live on two 7-segment displays on the board.

---

## Hardware Specifications

* **Board:** Terasic DE10-Standard
* **FPGA:** Cyclone V SoC
* **Language:** Verilog HDL
* **Clock:** 50 MHz (`PIN_AF14`)
* **Display:** 640×480 @ 60 Hz VGA
* **Synthesis:** Intel Quartus Prime 20.1
* **Controls:** 4× Push-buttons
* **Score:** `HEX0` + `HEX1` (7-segment displays)

## System Architecture & Modules

### `Clks_Generator.v` (The Timekeeper)
The 50 MHz master clock is divided down into two essential signals:
* **25 MHz `VGA_clk`:** Created using a toggle flip-flop (the simplest possible divider) to provide the exact pixel clock required for the VGA.
* **~8 Hz Game Update:** A toggle signal generated by a counter (`UPDATE_MAX`) where `50,000,000 ÷ 3,000,000 ÷ 2 = ~8 Hz`. The rest of the design detects any edge on this signal as a game tick. Speed can be tuned by halving `UPDATE_MAX` to double the speed, or doubling it to halve the speed.

### `snake.v` (Top Level & Colour Logic)
This glue module instantiates all sub-modules and contains the critical combinational logic for pixel coloring. There is no frame buffer; on every 25 MHz clock edge, the current `xCount`/`yCount` tells the logic exactly one pixel, and the color is decided before the scanner moves on:
* **Snake Head:** Red + Green = Yellow (stands out from the body).
* **Snake Body:** Green only (bright, easy to track).
* **Border:** Green only (same as body, clear wall).
* **Apple:** Red only (obvious target).
* **Background:** Blue only (calm, contrasts well).
* **Game Over:** Full red (floods entire display).

### `Controller.v` (Reading the Buttons)
Translates the four active-low push-buttons into a 3-bit direction code. The KEY buttons on the DE10-Standard read 0 when pressed and 1 when released due to pull-up resistors.
* `KEY[0]` → UP (`3'b001`)
* `KEY[1]` → DOWN (`3'b011`)
* `KEY[2]` → LEFT (`3'b010`)
* `KEY[3]` → RIGHT (`3'b100`)
* `KEY[3]` held → RESET (`3'b111`)
This module is purely combinational (no clock or registers), so the direction code updates in nanoseconds the moment a button changes.

### `collision.v` (The Heart of the Game)
The largest module running two parallel processes simultaneously:
* **Process 1: Game State Machine (~8 Hz):** Shifts the body array tail-first (`Segment 31 ← 30 ← … ← 1 ← head`). The head moves by predicting the next position (adding/subtracting a 10px `SEG_SIZE`) before the clock edge to avoid detecting collisions one tick late. It checks for overlaps with the apple box (to grow size, increment score, and pulse reroll), the walls (outside `x:31–599` or `y:31–439`), or its own body.
* **Process 2: Pixel Membership (25 MHz):** For every pixel the VGA scanner visits, 32 comparisons run in parallel as combinational logic, evaluating within the 40 ns pixel clock window to determine if the pixel belongs to the snake's body.

## Game Flow

1.  **Start:** When `SW[0]` goes HIGH, the reset condition de-asserts. The snake initializes at position (300, 240) with a length of 3 segments, moving RIGHT.
2.  **Display:** The VGA scanner continuously sweeps at 25 MHz without interruption; the game smoothly transitions from a blank frame to the active game.
3.  **Movement:** Every ~125 ms, the update signal toggles, causing the state machine to shift the body array and move the head one 10-pixel step. 
4.  **Input:** Button presses cause an immediate direction change in the combinational controller, which takes effect on the very next game tick for an instantaneous feel.
5.  **Scoring:** If the head lands on an apple, the snake size and score increment, and the LFSR is sampled instantly to jump the apple to a new pseudo-random position.
6.  **Game Over:** If the head exits the play field or matches a body segment, the `game_over` signal latches HIGH, flooding the screen red and halting the state machine ticks until `SW[0]` is toggled.
